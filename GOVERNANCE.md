# Governance

MaintainerKit currently uses a lightweight maintainer-led governance model.

## Maintainer

`fishfillet-krm` is the founding and primary maintainer. The maintainer sets project direction,
reviews contributions, manages releases, and is accountable for security and compatibility
decisions.

## Decision Making

- Routine fixes and documentation improvements are decided through public issues and pull
  requests.
- Public API, security boundary, GitHub write automation, and release changes require explicit
  maintainer approval.
- Significant design changes should begin with an issue that records goals, alternatives, and
  compatibility risks.
- Decisions should favor small, reversible changes and transparent evidence.

## Becoming a Maintainer

Repeat contributors may be invited after demonstrating sustained, constructive participation,
sound review judgment, and respect for the project's safety model. Maintainer access is never
granted solely for contribution volume.

## Conflicts

The primary maintainer makes the final decision when consensus cannot be reached. The reasoning
should be documented publicly unless security or privacy requires confidentiality.
